发表于 2017-05-18

Homebrew安装

  • /usr/bin/ruby -e “$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/master/install)"

##GIT安装

  1. git version 判断是否安装git
  2. Mac可以通过Homebrew安装，brew install git
  3. Linux可以通过yum install git-core
  4. Ubantu可以通过apt-get install git
  5. Windows可以通过git.exe安装，配置git的环境变量即可。

GIT与SVN区别

  • Git是分布式的版本管理系统，而SVN是集中式的版本管理系统。
  • 在Git 中的绝大多数操作都只需要访问本地文件和资源，不必联网就可以看到所有的历史版本记录，而SVN 却需要联网。
  • SVN 断开网络或者断开VPN就无法commit代码，但是Git 可以先commit到本地仓库。
  • Git 克隆一个完整项目的速度非常快，SVN 非常慢。我们以克隆一份拥有五个分支的完整项目以及版本库来说，SVN是同时复制5个版本的文件,也就是说重复五次同样的动作。而Git 只是获取文件的每个版本的元素，然后只载入主要的分支(master)在我的经验，克隆一个拥有将近一万个提交(commit)，五个分支，每个分支有大约1500个文件的 SVN，耗了将近一个小时！而Git只用了区区的1分钟。
  • 其中最重要的区别是在于Git 上的分支远比SVN上的强大。
  • SVN的分支是一个项目的目录，是整体的，你修改一个分支，别人需要更新这个项目上你修改过的分支，且还要联网更新，不像Git一样，分支随意切，这也和Git只关心文件数据的整体是否发生变化，SVN只关心文件内容的具体差异有关。

GIT原理

  • git使用的是SHA-1哈希算法，计算数据和校验和，通过对文件的内容或目录的结构计算出一个SHA-1的哈希值后，作为指数的字符串

GIT通信原理

  • Git采用的是 SSH 加密原理，是采用了非对称加密中的RSA算法

GIT通信公钥

  1. cd ~/.ssh目录
  2. ssh-keygen -t rsa -c “邮箱地址” （这步是在ssh目录中，命令会产生一对秘钥，创建公钥私钥）
  3. Enter file in which to save the key (/Users/you/.ssh/id_rsa): [Press enter] // 推荐使用默认地址
  4. Enter passphrase (empty for no passphrase): //此处点击 Enter 键即可，也可以填写密码，填写密码后每次使用 SSH 方式推送代码时都会要求输入密码，由于这个 Key 也不是用于军事目的，所以也无需设置密码。
  5. 本地会生成id_rsa（私钥），id_rsa.pub（公钥匙）, vi/vim 访问id_rsa.pub，复制里面的信息
  6. 在Git服务器中，仓库设置里导入公钥信息，就可以完成与服务器的GIT交互了
  • 注意：有多个账户，可以用ssh-keygen -t rsa -C “your_secondemail@email.com“ -f ~/.ssh/second_rsa，然后ssh-add ~/.ssh/id_rsa ~/.ssh/second_rsa添加第二个公钥即可

GIT HEAD概念

  • head 指针指向表示当前活跃的节点（分支）。

实用场景

  1. 通过gitlab，github，coding，bitbucket等git服务器平台，创建Git远程仓库，得到地址
  2. 设置好RSA的公钥，私钥，粘贴私钥 git clone 项目地址，把项目拷贝到本地仓库
  3. 开始项目架构
  4. git add 修改过的文件（一般为整个项目工程）
  5. git commit -m “修改的信息1. 2. 3.”，提交代码到本地仓库
  6. git push ，提交主体架构到远程仓库，默认提交至远程仓库的master主干上（主分支），如果当前项目为分支的话，提交代码到的是远程分支
  7. 本地创建分支git branch xxx ，命名dev，打上Tag，提交分支到远程分支 git push origin 分支名，假设有N个人，提交分支到项目中。
  8. 项目管理员，收到分支后，会先 clone 该分支到本地仓库，然后，检查写好的代码无误后，切换本地仓库的分支到mater(eg. git branch 查看分支信息—>git branch master 切换分支到master )，使用命令git merge 分支名（eg. git merge dev），在本地仓库，此时已经完成分支的合并。
  9. 将本地仓库的主分支master 推送远程的主分支master。（eg. 只有项目拥有者才有权限推送本地master到远程master)

常见问题

还没提交代码，不小心按了 Discard All Change（放弃所有操作）

  • 原因：误操作
  • 解决方法：假设你有每天Commit的习惯，还可以拯救一部分代码，假设你从头到尾都没进行Commit的话，本地的代码就真的无法挽回了。可以通过git log 查看提交记录，然后 git -reset hard 提交的id来回滚到几天前的代码。
  • 解决方法2: 1.退出XCode 2.用文本编辑器打开被修改了的文件 3.文本编辑器下，”文件”–”复原到”–”浏览所有版本…” 4.在弹出的窗口中选择你需要的版本恢复。
  • 困难模式：比如有两次提交，98abc5a，b7057a9，我们假设回滚到98abc5a第一次提交的时候，这时候用git log ，这看起来我们是丢掉了我们第二次的提交，本地的修改也消失了，没有办法找回来了。但是 reflog 就是用来解决这个问题的。简单的说，它会记录所有HEAD的历史，也就是说当你做 reset，checkout等操作的时候，这些操作会被记录在reflog，使用 git reflog，查看记录。

最终工程的代码里面，把自己的代码合并没了

  • 原因：如果自己新建的分支，无意删除了别人写好的代码，然后合并到master 就会导致master的代码也没有，因为最终项目管理员只会合并一个人的代码，所以要确保其他在代码合并的时候是否争取；
  • 解决方法：git pull 拉取分支代码确保是最新代码，或者建立分支的时候，从远程clone别人分支代码。

git merge合并代码后冲突

  • 原因：两个用户修改了同一文件的同一块区域
  • 解决方法：手动解除

revert与reset的区别

  • git revert是用一次新的commit来回滚之前的commit，git reset是直接删除指定的commit。

git merge合并代码后，编译工程，Stroyboard报错

  • 原因：两个用户修改了同一个SB文件
  • 解决方法：比较报错的SB文件的XML文件

操作核心

  • 先拉后推（pull –> push ）

MVC/MVVM/MVP误区扫盲

MVC是框架还是设计模式？

  • MVC是一种软件架构模式，它包含了很多的设计模式，最为密切是以下三种：Observer（观察者模式，如KVO），Composite（组合模式）和Strategy（策略模式）
  • MVC模式基本思想：数据，显示，处理相分离，数据（Model）层负责数据管理，视图（View）负责数据显示，控制器（Controller）负责业务逻辑和响应策略。
  • 使用这种架构模式，能够使程序 1.耦合性低 2.重用性高 3.部署快 4.可维护性高 5.有利于软件工程管理 6.有利于团队开发
  • 根据MVC架构模式思想 设计出对应的框架代表，如：Spring MVC，asp.net

##MVP的理解

  • 是不是Apple’s MVC事实上就是MVP？并不是，回想一下在MVC中View和Controller是紧密耦合的，然而，MVP的中介——Presenter与View Controller的生命周期没有任何关系，并且很容易模拟View，所以Presenter中没有任何布局代码，但是它却负责用数据和状态更新View

  • MVP是第一个揭示出组装问题（assembly problem）的架构模式，而出现这个问题的原因是它有三个实际上独立的层。

  • 首先MVVM的思想是由MVP演化而来的，就MVP来说，UIViewController的子类实际上就是视图（Views）不是展示器（Presenters），这点区别带来了极好的可测试性，而花费代价是开发速度，因为你不得不手工进行数据和事件绑定

MVVM理解

  • MVVM是为了实现纯净的MVC模式，主要目的是为了剥离View和Model的直接交互，让Controller实现真正的仅仅数据交换协调功能，能够实现这个目的，就是让View和Controller之间多了一个ViewManger的交互，model和Controller之间多了一个ViewModel的存在，从而剥离开V和Model的直接关系（如KVC这种View，Model赋值，tableview的cell和Model直接交互），其中RecatObjc/RxSwift这种响应式编程框架，比较适合MVVM的模式，刚好就可以实现MVVM的形式，也可以通过自己封装ViewManger和ViewModel实现MVVM的架构。

为什么在乎架构的选择？

因为如果你不这样做，终有一天，你在调试一个拥有着数十个不同方法和变量（things）的庞大的类文件时，你会发现你无法找到并修复此文件中的任何问题。自然地，也很难把这个类文件当做一个整体而熟稔于心，这样你可能总是会错过一些重要的细节。如果你的应用已经处于这样的境况，很有可能是这样：

  • 这个类是UIViewController的子类。
  • 你的数据直接存储在UIViewController中。
  • 你的UIViews什么都不做。
  • Model是哑数据结构（dumb data structure ）。

让我们定义一下一个好的架构应该有的特点：

  1. 能把代码职责均衡的解耦到不同的功能类里。（Balanced distribution of responsibilities among entities with strict roles.）
  2. 可测试性（Testability usually comes from the first feature.）。
  3. 易用、维护成本低（Ease of use and a low maintenance cost.）。

iOS SDK(动态库/静态库)

发表于 2016-05-18

基本概念

库概念

* 开源库(eg.AF,SDWeb,能看到.m实现)
* 闭源库:1.静态库 (形式:.a/.framework) 2.动态库 (形式:.tbd || .dylib(之前的叫法) /.framework)
制作SDK方法 
1. .a静态库制作,Xcode模版,创建Static Library: 最终生成 .h && .a 两个文件。(eg:sina,wechat)
2. .framework静态库制作,最终生成.framework文件,推荐使用。 => (eg:baidu,renren)
3. .workspace的方式,多工程联编。(eg:FaceID)

优缺点

1.静态库
* 链接时完整地拷贝至可执行文件中，被多次使用就有多份冗余拷贝。提高了代码的复用及核心技术的保密程度。    
* 编译后的执行程序不需要外部的函数库支持，因为所有使用的函数都已经被编译进去了。当然这也会成为他的缺点，因为如果静态函数库改变了，那么你的程序必须重新编译。
2.动态库
* 可以将最终可执行文件体积缩小.
* 程序运行时由系统动态加载到内存，节省内存。
* 动态函数库的改变并不影响你的程序，所以动态函数库的升级比较方便。

区别

* 静态库:链接时会被完整的复制到可执行文件中，被多次使用就有多份拷贝
* 动态库:链接时不复制，程序运行时由系统动态加载到内存，系统只加载一次，多个程序共用（如系统的UIKit.framework等），节省内存。

总结



    

  • 静态库和动态库是相对编译期和运行期的
    • 

  • 静态库在程序编译时会被链接到目标代码中，程序运行时将不再需要改静态库；
    • 

  • 动态库在程序编译时并不会被链接到目标代码中，只是在程序运行时才被载入，因为在程序运行期间还需要动态库的存在。
    • 





iOS8之后的变动
    

  • iOS8 之前，苹果不允许第三方框架使用动态方式加载。
    • 

  • iOS8 开始允许开发者有条件地创建和使用动态框架(Building->Linking->Mach-O->Static Library)，这种框架叫做 Cocoa Touch Framework
    •

  • 原因:含有Category
    •

    • 





添加静态库后,编译报错 “有重复定义的符号(duplicate symbol )”

基本概念
    

  • 可以把沙盒理解成，每个App在iOS 系统硬盘中的总文件夹，这个沙盒文件夹，它采用的沙盒安全机制，即使你在沙盒里弄病毒之类的，都不会影响到系统级别的更改，沙盒文件只有访问自己，不能访问其他App或者系统的权限，应用程序只能访问该目录下的文件夹！
    • 



实际例子
    

  • Windows系统一样，中了病毒或者木马会一直消耗内存，直到内存没有了才宕机
    • 

  • Apple的应用程序在内存消耗过高时会收到内存警告，如果不及时处理的话应用程序会自动退出
    • 

  • 沙盒的本质就是一个⽂件夹，名字是随机分配的，所以每次的文件路径不一样
    • 



沙盒文件分类


Document




    

  • 这个目录用于存储用户数据，可被iTunes备份，可以通过iCloud自动备份，一般用来存放应用中建立的文件, 如数据库文件, 或者程序中浏览的数据, 如果进行备份将会备份此文件夹内容
    • 





Library：包含两个目录




    

  • Preferences 目录：包含应用程序的偏好设置文件。您不应该直接创建偏好设置文件，而是应该使用NSUserDefaults类来取得和设置应用程序的偏好.
    

    • 

  • Caches 目录：用于存放应用程序专用的支持文件，保存应用程序再次启动过程中需要的信息。可以重新下载或者重新生成的数据应该保存在/Library /caches目录下面。举个例子，比如杂志、新闻、地图应用使用的数据库缓存文件和可下载内容应该保存到这个文件夹。
    

    • 

  • 可创建子文件夹，用来放置您希望被备份但不希望被用户看到的数据。该路径下的文件夹，除Caches以外，都会被iTunes备份。
    

    • 





Temp




    

  • 存放应用运行时产生的一些临时数据和文件，当应用程序退出、系统磁盘空间不足、手机重启时，都会自动清除该目录的数据。无需程序员手动清除该目录中的数据，iTunes、iCloud备份时不会备份此目录
    • 





MyApp.app




    

  • 包含程序中的nib⽂件、图片、音频等资源。
    • 

  • iOS8之后，.app文件被移动到了Container——>bundle——>Application文件夹下
    •

思考


为什么沙盒路径就安全了？




    

  • 应用程序不能翻过自己的围墙去访问别的存储空间的内容。(不能破坏系统文件，或者其他应用文件)
    • 

  • 应用程序请求的数据都要通过权限检测，假如不符合条件的话，不会被放行。
    • 





为什么用的是Search的API，去寻找路径？




    

  • 沙盒文件是系统随机分配的，每次应用打开，系统会随机分配，路径就发生改变，没有绝对的路径
    • 





wechat切换用户的时候，取出的是对应用户的数据？




    

  • 创建对应用户在沙盒路径下的对应的路径，在应用启动的时候判断用户是否为登录状态，如果不是登录状态，就不设置对应的文件，如果用户登录了，创建该用户对应的文件路径，读取也是改用户的路径
    • 





如果更新了APP后，相关的沙盒文件路径作修改了怎么办？




    

  • 设置一个控制器，根据版本号判断，加上之前的逻辑判断，判断是否进行一些沙盒文件的删除和迁移 
    • 





iOS版本升级过后Core Data 的数据库如何迁移？




    

  • 每次版本升级就删除本地数据库（太傻）
    • 





沙盒写入文件，如果路径相同，是否会覆盖之前的文件？




    

  • 会覆盖之前的文件，如果需要追加数据，在同一个文件上的话，

COCOA框架理解
    

  • COCOA框架是苹果提供编程框架(Cocoa本身是一个框架的集合,它包含了众多子框架)，使用这个框架可以实现基于iOS系统的APPLE应用的开发。
    

    • 

  • COCOA框架的子框架（如：Foundation.framework，UIKit），提供了iOS系统，绝大部份的底层硬件的类，抽象类，网络类，图像类，等封装。
    

    • 

  • Foundation 是框架的基础，和界面无关，其中包含了大量常用的API；后者是基础的UI类库，以后我们在iOS开发中会经常用到。这两个框架在系统中的位置如下图
    

    • 

  • 框架就是由许多类、方法、函数、文档按照一定逻辑组织起来的集合，以便使我们的开发变得更容易。Foundation，顾名思义，就是为所有iOS应用程序开发奠定基础的框架，它不仅包含了如结构体、数组、字典、集合等对象集合，也包含了内存管理、处理基础文件系统、存储对象等
    

    • 

  • COCOA框架示意图

    从Cocoa Touch到Core OS下面四层包含了开发iOS应用程序所用到的所有API（第三方框架也是基于这几个层的

    •

    • 

  • Cocoa Touch包含Foundation和UIKit框架，可用于开发iPhone OS 系统的应用程序
    • 

  • Cocoa是Mac OS X的开发环境，Cocoa Touch是 iPhone OS的开发环境
    •

库与框架的区别
    

  • 库的英语为Library（Lib），框架的英语为Framework
    • 

  • 类库/函数库：开发者在使用库的时候，只需要使用库的一部分分类式函数，然后继续实现自己的功能
    • 

  • 框架：是为了解决一个（一类）问题而开发的产品，框架用户一般只需要使用框架提供的类或函数，即可实现全部功能，可以说框架是库的升级版
    • 



形象比喻
    

  • 框架相当于一部组装好的电脑
    • 

  • 库就如自己的组装的电脑。库为我们提供了很多部件，我们需要自己组装，如果某个部分未提供，我们也可以自己做
    • 



彼此之前的区别
    

  • 库：工具箱
    • 

  • 框架：一套解决方案，重写代码就能够实现全部的功能
    • 

  • 架构：是高度抽象的需求（相当于房子的房型图）
    • 



设计模式与架构的区别
    

  • 设计模式 ：思想
    • 

  • 架构：方案，包含是设计模式和已存的框架与实际的业务需求
    • 

  • 框架：就是软件成果，能实现代码的重用，帮你实现一个类型的业务需求
    •
